\chapter{Methodology}
\label{chap:met}

This chapter describes the design of the proposed performance monitoring system. Section~\ref{sec:req} lists the requirements that influenced the design choices. Section~\ref{sec:arch} gives a high-level overview of the architecture and shows how the components work together. Sections~\ref{sec:sdk},~\ref{sec:backend} describe the details of the mobile SDK and the Backend. Section~\ref{sec:detection} discusses the regression detection methods that were considered and highlights the selected one.

\section{System requirements and constraints}
\label{sec:req}

Since the SDK (Software Development Kit) runs on real user devices, it must be lightweight so that it doesn't affect the user experience of the host application. Moreover, the system must be adaptable to different device performance levels. Finally, regression detection must flag sustained degradation between consecutive releases rather than reacting to momentary spikes.
\section{High-Level System Architecture}
\label{sec:arch}

The system consists of four components: the mobile SDK (client), the Backend, the frontend, and the regression detection engine. Figure~\ref{fig:system_architecture} shows how they are connected.

The mobile SDK is a library that developers integrate into their Android applications. It collects performance metrics on users' devices and sends them to the backend for further analysis.

Metrics from user devices arrive at the backend, which stores them efficiently and serves them to the frontend through REST APIs. The backend is also where user authentication and project management happen.

The frontend is a web application for developers. Through it they manage their projects, explore the collected metrics, and look over any regressions that were found. It pulls all of this from the backend through the same REST APIs and lays the data out in a readable form.

The regression detection engine continuously analyses the incoming metrics to find signs of performance degradation. If such a regression is detected, it can notify developers in a messaging app.

\section{Mobile SDK Design}
\label{sec:sdk}

\subsection{Metrics Selection and Collection}

The metrics were selected based on the literature review in Chapter~\ref{chap:lr}. The guiding criterion was that the collected metrics should reflect how the user actually perceives the application, rather than only describe low-level system behaviour. As discussed in the review, performance metrics fall into two kinds: resource metrics, such as CPU and memory usage, that explain where a problem originates, and user-perceived metrics, such as frame time, startup time, and interaction delay, that show how strongly that problem affects the user. The two are linked, because resource pressure usually surfaces as a slower or less responsive interface. The SDK therefore collects user-perceived metrics as the primary signal of experienced performance, together with resource metrics that help explain the cause of a perceived slowdown.

These metrics are collected in two ways: as continuous metrics and as discrete events. Continuous metrics, such as CPU usage and frame time, are sampled at regular intervals. Discrete events, such as crashes, ANRs, and app startups, are recorded immediately when they happen. This approach allows the system to capture both the ongoing performance of the application and the rare but critical events that strongly affect user experience.

\subsection{UI performance metrics}

For UI performance, the SDK tracks frame time, FPS, startup time, interaction delays, and ANR events.

\textbf{Frame time} is one of the most important metrics for measuring UI responsiveness. It represents the time needed to render a single frame of the user interface. The target frame time depends on the display refresh rate of the device. For example, on a 60 Hz display, the frame time should stay below approximately 16.67 ms in order to keep animations and interactions smooth \cite{AndroidVitals}.

Frame time is measured by recording the timestamps of two consecutive frames and calculating the difference between them. If $t_i$ and $t_{i-1}$ are the timestamps of two consecutive frames, the frame time is calculated as follows:
\begin{equation}
    FrameTime(ms) = \frac{t_i - t_{i-1}}{10^6}
\end{equation}

\textbf{Frames per second (FPS)} is a related metric that is derived directly from the frame time. It is calculated as:
\begin{equation}
    FPS = \frac{1000}{FrameTime(ms)}
\end{equation}

\textbf{User interaction delay} measures the time between a user action (e.g. tap on a screen or a swipe) and the first visible response of the application. It is obtained by recording the timestamp of the user event and the timestamp of the first frame that was rendered after the event was processed. The delay is then calculated as:
\begin{equation}
    InteractionDelay(ms) = t_{\text{response}} - t_{\text{event}}
\end{equation}

\textbf{Startup time} shows how long the application takes to become visually responsive after it has been launched. It is defined as the time difference between the moment the application starts and the moment when the first fully rendered frame is drawn on the screen.

\textbf{ANR (Application Not Responding) events} happen when the application is unable to handle user input for a long period of time. This is a critical metric, because ANRs directly affect user experience and can lead to the application being terminated by the operating system. ANR events are detected by monitoring the main thread. If the main thread remains blocked for more than a predefined threshold (for example, 5 seconds), an ANR event is recorded.

\subsection{CPU usage}

CPU usage is measured from the CPU time consumed by the application process. The process CPU time is read at the start and end of each sampling interval, and the usage is the increase in process CPU time divided by the elapsed wall-clock time of the interval. A system-wide alternative, which would divide the process CPU time by the total CPU time of all processes, would give a value normalised to the device capacity, but it requires reading \texttt{/proc/stat}, which modern Android versions restrict for security reasons. The process-level measure is used instead, as it relies only on the public platform API and adds little overhead. Because the application can run on several cores at once, this value can exceed 100\%, and it depends on the core count and clock speed of the device. Cross-device comparison is therefore not made by the formula itself, but by the cohort stratification and the release-pair comparison described in Section~\ref{sec:detection}. The method may also miss short spikes that occur between two samples, but this is acceptable here, because the goal is to detect sustained degradation rather than momentary load.

Let $\Delta CPU_{process}$ be the increase in process CPU time and $\Delta t_{wall}$ the elapsed wall-clock time over the same interval. The CPU usage of the application is calculated as:
\begin{equation}
    CPU\% = \frac{\Delta CPU_{process}}{\Delta t_{wall}} \cdot 100
\end{equation}

\subsection{Memory usage}

There are also several methods for measuring the memory used by an Android application. In this work, memory usage is measured using the Proportional Set Size (PSS) of the application process. The PSS value represents how much physical memory is effectively used by the application, including a proportional share of the memory that is shared with other processes.

The memory usage is reported in megabytes and is calculated from the PSS value in kilobytes as:
\begin{equation}
    MemoryUsage(MB) = \frac{PSS(KB)}{1024}
\end{equation}

This metric is important, because it helps to detect memory leaks and excessive memory consumption, which are frequent causes of application slowdowns and of termination by the operating system.

\subsection{Data Aggregation and Transmission}

The metrics collected on the device are not sent immediately. Instead, they are first stored in a local buffer and then transmitted in batches. Batching was chosen because the radio interface stays in a high-power state for several seconds after each transmission, so frequent network calls are disproportionately energy-intensive~\cite{Balasubramanian2009, LiHttpEnergy}. Sending metrics in batches reduces this cost and also lowers the load on the backend.

A batch is sent when one of two conditions is satisfied: either the number of metrics in the buffer reaches a predefined value $N$, or the time since the last transmission exceeds a time window $W$.

\section{Backend Design}
\label{sec:backend}

The backend has four jobs: it receives metrics from the devices, authenticates users, exposes APIs for the frontend, and runs the regression detection algorithms.

When a batch of metrics arrives from the SDK, the backend validates the request, enriches each record with device cohort information, and writes the data into the time-series database.
After that, the frontend can query the stored metrics through REST endpoints, and the regression detection engine runs in the background to analyse new data as soon as it arrives.

\subsection{Database Design}

The backend relies on two kinds of database. A time-series database stores the performance metrics, and a relational database stores the rest: the users, their projects, and the regressions found for those projects. The time-series store was chosen because it handles large volumes of time-stamped data efficiently, whereas the relational store fits the structured, interconnected records that describe accounts and their projects.

The relational side holds four entities, shown in Figure~\ref{fig:postgres_schema}. A \emph{user} owns one or more \emph{projects}, each identified by its Android package name. A project, in turn, owns a catalogue of \emph{version releases} and its detected \emph{regressions}. The \emph{version releases} entity records each application version observed for a project, together with when it was first seen and how many samples it has accumulated. This catalogue is refreshed periodically from the time-series store and is used to list the releases of a project. The \emph{regressions} entity records each confirmed regression: the affected metric, screen, and device cohort, the baseline and current versions with their P95 values, and a status that marks it as open or closed. This ownership structure also enforces tenancy isolation, since every query can be scoped to the projects that belong to the requesting user.

    \centering

    \caption{Relational data model used by the backend}
    \label{fig:postgres_schema}

Performance metrics are kept separately in the time-series store and are not modelled as relational entities. For this purpose the system uses ClickHouse, a column-oriented database. A column-oriented layout fits the metric workload well: ingestion is append-only and write-heavy, while analysis reads only a few columns at a time and aggregates them over large ranges, such as computing a percentile per version~\cite{schulze2024clickhouse}. Each metric record carries a value, its timestamp, and the dimensions along which it is later grouped, namely the project, the metric type, the screen, the device cohort, and the application version. The data is organised so that a query restricted to a recent period and to a single project and metric scans only the relevant subset, which keeps the regression-detection queries efficient as the volume of data grows.

Metrics are also retained only for a limited period of time. Older data is removed automatically, which keeps the working set relevant and reduces storage cost. The physical layout of both stores, including table definitions and indexing, is described in Chapter~\ref{chap:impl}.


\subsection{Data Partitioning and Cohorts}

Before applying any detection model, the incoming data must be grouped into meaningful subsets. Comparing high-end devices directly with low-end devices could easily produce false positives, because the natural hardware limits of a low-end device might be interpreted as a software regression. To avoid this, the data is partitioned along three dimensions: the metric type (for example, CPU usage or frame time), the application screen, and the device performance cohort (Low, Medium, or High). The detection algorithms are then executed independently on each subset.

\subsection{Performance Regression Detection}
\label{sec:detection}

The regression detection pipeline compares consecutive application releases using a release-pair P95 shift. Samples are grouped by version code rather than by a fixed time window, and each release is compared against the one that precedes it. Fixed absolute thresholds were rejected because each one must be calibrated by hand for every metric and application, and a fixed limit does not move as baseline performance shifts over time. A mean-based comparison was avoided as well, since the mean is easily pulled by outliers and copes badly with the heavy-tailed, non-normal distributions that frame time and startup time usually follow. Heavier methods such as ARIMA forecasting or machine-learning models were ruled out too, because they need labelled training data or careful tuning that a lightweight production system cannot justify.

Detection runs only on user-perceived metrics, such as frame time and startup time, because a regression matters when the user actually feels it. Startup time, the delay before the application becomes responsive after launch, is treated as a regression target alongside frame time. Resource metrics such as CPU and memory usage are still collected, but they serve to explain the cause of a perceived slowdown rather than to raise alerts, so a fluctuation in resource use that the user never notices does not trigger a false alarm.

A version is only eligible for comparison after it has accumulated a minimum number of samples. This maturity gate prevents comparisons based on a handful of early measurements, where a single noisy observation could shift the P95 far enough to cross the threshold and produce a false alert.

For each mature version group, the 95th percentile ($P_{95}$) of the metric is computed. A regression is reported when the relative increase from the baseline version to the current version exceeds a configurable threshold:
\begin{equation}
    \frac{P_{95}(\text{current}) - P_{95}(\text{baseline})}{P_{95}(\text{baseline})} > Threshold
\end{equation}
Using $P_{95}$ rather than the mean focuses detection on the worst user experiences rather than the average, and makes the result robust to skewed distributions~\cite{Nguyen2012, DeanTailAtScale}. The threshold expresses a directly interpretable criterion: a given percentage of degradation. Unlike the absolute, per-metric limits used by the threshold-based tools discussed in Chapter~\ref{chap:lr}, it is a relative value that compares each release against the previous one, so a single default (15\%) applies across all metrics, screens, and device cohorts. The developer therefore does not have to choose or calibrate a threshold for every metric and application. The system uses this default out of the box. This avoids the main drawback of fixed absolute thresholds, where each limit must be set and maintained by hand. The pipeline runs on production data without any labelled examples, and its core comparison is expressed as a single aggregation query, making it computationally efficient and suitable for continuous execution. The detection rule is deliberately kept simple and interpretable: a single comparison against a configurable percentage.

Because $P_{95}$ is taken over the whole version group, it ignores anything that affects fewer than five percent of the samples. A momentary spike is a handful of high values among the thousands collected from many users over the lifetime of a version, so it sits above the 95th percentile and does not move $P_{95}$. The value shifts only when the degradation becomes frequent enough to reach the top five percent of the samples, which is a sustained problem rather than a spike. This is what lets the rule flag sustained degradation while ignoring momentary spikes.

\section{Summary}

This chapter presented the methodology behind the proposed performance monitoring system. The design was driven by the requirements from the literature review, with an emphasis on low overhead, energy efficiency, and robustness across devices. The mobile SDK collects metrics on the user's device and sends them to the backend in batches. The backend stores them, exposes them through an API, and runs rule-based regression detection on user cohorts. Together, these parts give developers automated insight into how their applications perform in production. The next chapter turns this design into a working system and describes the technical problems that came up along the way.
