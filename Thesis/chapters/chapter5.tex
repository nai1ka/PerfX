\chapter{Evaluation and Discussion}
\label{chap:eval}

This chapter evaluates the system against the research questions stated in Chapter~\ref{chap:intro}. Section~\ref{sec:eval_setup} describes the validation setup. Section~\ref{sec:eval_overhead} measures the SDK overhead, addressing the first research question about low-overhead monitoring. Section~\ref{sec:eval_regression} presents an end-to-end test of the whole system, addressing the second and third research questions on automatic regression detection and the role of device performance cohorts in detection. Section~\ref{sec:eval_discussion} summarises the findings, and Section~\ref{sec:eval_limitations} discusses the limitations.

\section{Validation Setup}
\label{sec:eval_setup}

The experiments were carried out on one physical Android device and three emulator configurations, each defined by a different amount of RAM and a different number of CPU cores, so that the results cover all three performance cohorts defined in Section~\ref{sec:detection}. Table~\ref{tab:eval_devices} lists each device configuration together with its hardware characteristics, its Android version, and the cohort assigned by the backend. Individual scenarios may run more than one emulator instance of a given configuration, and the per-scenario counts are reported in the corresponding subsection.

Two separate applications were used across the evaluation. The overhead evaluation used Cheddar~\cite{cheddar}, an open-source Android client for the Hacker News website, representing a realistic workload of network requests, list rendering, and in-app navigation. Two builds with different version codes were compared so the detector treated them as consecutive releases.

\section{SDK Overhead Evaluation}
\label{sec:eval_overhead}

The SDK overhead was measured under a single Cheddar workload, built four times under distinct product flavors. The baseline build (no monitoring SDK) defines the reference. PerfX is compared against two widely used industry SDKs: Firebase Performance Monitoring \cite{FirebasePerf} and Datadog RUM. Each build was installed in turn, cold-started twenty times to measure startup time, then left running for ninety seconds while an automated scrolling script exercised the story feed and CPU and PSS memory were sampled every second.

PerfX added 19.5 percentage points of CPU usage, 6.3~MB (+3.9\%) of resident memory, and 276~ms (+15.9\%) of cold-start time. Firebase Performance Monitoring and Datadog RUM behaved similarly to each other, but produced overhead an order of magnitude higher: roughly 78 percentage points of CPU and around 58~MB of memory, between nine and ten times the PerfX memory cost and around four times the PerfX CPU cost. The cold-start penalties of the three SDKs were closer to one another, at 177--276~ms, since cold-start time is dominated by one-shot class loading and SDK initialisation rather than steady-state work.

The PerfX CPU cost remains the lowest of the three SDKs compared here, and is comparable to the approximately 6\% reported for the Panappticon tracing system~\cite{Panappticon} once Cheddar's higher baseline activity is taken into account. Together with the small steady-state memory overhead and a moderate one-time cold-start cost, these results support the low-overhead requirement defined in Section~\ref{sec:req}.

\section{End-to-End System Testing}
\label{sec:eval_regression}

This section tests the full system end to end. A synthetic regression is injected into a build by adding extra CPU work per rendered frame, which raises frame time under load. The build then runs on real devices, and the full system, from the SDK through the backend and detector to the dashboard, is checked for the expected outcome. The detection rule is a deterministic threshold that flags a release when the relative P95 shift exceeds 15\%, so the goal here is to verify that the system applies the rule correctly, not to estimate a statistical detection accuracy.

The system is tested in three scenarios. The first injects a frame-time regression and follows it through the whole system. The second compares two builds with no code change and checks that no alert is raised. The third runs a cohort-localised regression on all cohorts and checks that the alert reaches only the affected one. The reported results describe the observed behaviour of the system on a controlled set of cases, not statistical estimates from production traffic.

\subsection{Experiment Setup}

Each scenario compared two builds under consecutive version codes, so the detector treated them as successive releases. The complete path was checked: the SDK collects the metric on the device, the backend ingests it, the detector compares the P95 of the two releases per metric, screen, and cohort, creates a regression record when the relative shift exceeds 15\%, and sends an alert. The regression scenarios use the demo application, and the no-change scenario uses Cheddar. Each build was exercised for the same fixed run duration in every scenario, so the two versions in every release pair contributed similarly sized samples to the detector, and every version satisfied the minimum sample-count gate. % TODO: confirm the exact per-build duration used across scenarios 1 and 3 (Scenario 2 uses three minutes per build)

\subsection{Scenario 1: Frame-Time Regression}

The first scenario introduces a frame-time regression and checks whether the system works correctly. The regression adds a fixed amount of CPU work to each rendered frame, which raises the frame-time metric. The SDK collected the frame-time samples on the physical device, the backend deployed on a remote server received them, and the dashboard plotted the two builds together. Figure~\ref{fig:frametime_comparison} shows the dashboard view, where the regression build reaches higher frame times more often than the baseline. The detector then compared the two releases and recorded a regression, which the dashboard lists on its regressions page in Figure~\ref{fig:detected_frontend}. The system then delivered the alert in Figure~\ref{fig:alert}, which reports the affected metric, screen, cohort, version pair, and the resulting P95 shift. Each step produced the expected result.

\subsection{Scenario 3: Regression on Low-Cohort Devices Only}
\label{sec:eval_fragmentation}

The third scenario runs a frame-time regression that only affects performance on low-tier devices. The scenario used three High-cohort emulators, three Medium-cohort emulators, and one Low-cohort emulator, so the Low cohort contributed fewer samples than the other two. The collected samples were analysed in two ways: first aggregated across all devices without cohort stratification, and then split by device cohort. The experiment was repeated four times to confirm that the outcome was stable. Table~\ref{tab:eval_tier_repeated} reports both the aggregated shift and the per-cohort shifts for each run.

\begin{table}[htbp]
    \centering
    \small
    \caption{Frame-Time P95 Shift Across Four Runs, Aggregated and Per Cohort}
    \label{tab:eval_tier_repeated}
    \begin{tabular}{ccccc}
        \toprule
        \multirow{2}{*}{Run} & \multirow{2}{*}{Aggregated shift (\%)} & \multicolumn{3}{c}{Per-cohort shift (\%)} \\
        \cmidrule(lr){3-5}
        & & Low & Medium & High \\
        \midrule
        1 & 8.4 & 27.0 & 9.1 & 5.2 \\
        2 & 7.6 & 25.8 & 8.4 & 4.7 \\
        3 & 9.2 & 28.3 & 9.8 & 5.9 \\
        4 & 8.1 & 26.4 & 8.7 & 5.0 \\
        \bottomrule
    \end{tabular}
\end{table}

Without cohort stratification, the aggregated shift stayed below the 15\% threshold in every run, so the detector raised no alert. With cohort stratification, the Low cohort crossed the threshold in every run and triggered an alert, while the Medium and High cohorts stayed below it. The same samples therefore produced an alert only when analysed by cohort.

This matches the real device population, where most active Android devices fall into the Medium and High cohorts and the Low cohort is a minority \cite{GooglePlayDeviceCatalog}. When the same samples are analysed without cohort stratification, the strong shift on the Low cohort is diluted by the larger Medium and High groups, and the aggregated shift stays below the threshold. Cohort stratification recovers this signal by comparing each cohort separately, so a regression confined to the minority Low cohort still produces an alert.

\section{Discussion of Results}
\label{sec:eval_discussion}

This section interprets the results in relation to the three research questions defined in Chapter~\ref{chap:intro}.

The first research question asks how Android application performance can be monitored with low runtime overhead. The results in Section~\ref{sec:eval_overhead} show that PerfX added 19.5 percentage points of CPU usage, 6.3~MB (+3.9\%) of resident memory, and 276~ms (+15.9\%) of cold-start time under a realistic Cheddar workload. The same scenario produced 77.6 percentage points of CPU and 59.4~MB of memory for Firebase Performance Monitoring, and 78.4 percentage points and 57.7~MB for Datadog RUM, so PerfX uses about four times less CPU and roughly nine times less memory than either industry SDK while collecting a comparable set of metrics. These costs are consistent with the background sampling, two-tier buffering, and batched transmission design described in Chapter~\ref{chap:impl}.

The second research question asks how to develop a system that automatically detects performance regressions introduced by new application releases. The pipeline groups metric samples by version code, metric, screen, and device cohort, applies a maturity gate, and then flags a regression when the relative P95 shift between two consecutive releases exceeds the fixed 15\% threshold. This approach requires no manual labelling and no per-metric threshold tuning, and it runs against incoming production data.

The third research question asks whether performance cohorts help detect regressions that affect only specific classes of Android devices. The injected regression degraded frame time only on weaker hardware. Compared within cohorts, it was flagged on the Low cohort and correctly left the Medium and High cohorts untouched. If all devices had been compared together, the shift on the Low devices would have been diluted by the unaffected Medium and High devices and could have fallen below the threshold, so the regression would have been missed. Cohort stratification therefore allows a regression confined to one device class to be detected and attributed to that class. On the realistic Cheddar workload with no code change, the relative shift stayed below 9\,\% on every metric, so the system raised no false alert. Because the detection rule is a deterministic threshold, this is a verification that the system applies it correctly.

\section{Limitations and Future Work}
\label{sec:eval_limitations}

The evaluation has several limitations.

First, the system was not deployed to real users. Overhead and detection were measured on a small fixed set of devices running one usage scenario, which does not capture the variety of real-world usage.

Second, emulators do not fully replicate the hardware behaviour of physical devices, so the emulator results for CPU and memory should be treated as indicative rather than conclusive.

Third, the device cohort is assigned using a simple rule based only on total RAM and CPU core count. Two devices with the same RAM and the same number of cores can still differ in real performance because of their chipset, GPU, or thermal behaviour, so this rule may place a device in the wrong cohort and weaken the cohort-based comparison.

% Fourth, the detection algorithm applies a single fixed threshold to the relative P95 shift regardless of the metric type. Different metrics have different natural variability, so the same threshold may be too sensitive for one metric and not sensitive enough for another.

Fourth, the detector is designed to identify release-to-release regressions. As a consequence, gradual performance degradation accumulated over multiple releases may remain undetected if no individual release exceeds the configured threshold.

These limitations point to several directions for future work. The most important next step is a real-world deployment on a live application with real users, which would test the system against naturally occurring regressions under varied hardware, usage patterns, and network conditions. The cohort classification could also be improved. Instead of relying on static hardware attributes such as RAM and CPU core count, cohorts could be derived from each device's observed performance, for example by grouping devices according to a lightweight on-device benchmark . Finally, the set of collected metrics could be extended with battery and network usage, which are also relevant to mobile user experience~\cite{Saraiva} but are not currently collected.
